# Plan: Build `commit-composer` (Claude Code plugin, revdiff-style TUI)

## Goal
A Go TUI plugin for Claude Code that lets the user mark commits with rebase actions (pick / reword / squash / fixup / drop / edit / reorder) in a split-pane interface, then emits a structured plan that Claude applies via `git rebase`.

UX target: same feel as umputun's `revdiff` (https://github.com/umputun/revdiff) - launches in a TUI, vim-style navigation, structured markdown output on exit, slash command entry point.

## Tech stack
- Go (latest stable, table-driven tests per repo conventions)
- bubbletea + lipgloss + bubbles for TUI
- Shell out to `git` for all git operations (do not use go-git, its rebase support is weak)
- Drive non-interactive rebase via `GIT_SEQUENCE_EDITOR` env var feeding a generated todo file

## Data model
```go
type Action int
const (
    Pick Action = iota
    Reword
    Squash
    Fixup
    Drop
    Edit
)

type Op struct {
    SHA        string
    Action     Action
    NewMessage string  // for Reword
    OrigIndex  int     // for detecting reorder
}

type Plan struct {
    Base string  // base commit SHA (immutable)
    Ops  []Op    // ordered; reorder = ops slice reordered
}
```

## Keybindings
- `j` / `k` / arrows: move cursor
- `J` / `K`: move highlighted commit down / up (reorder)
- `p` `r` `s` `f` `d` `e`: set action on highlighted commit
- `Space`: cycle action
- `R`: open modal to enter new message (for `reword`)
- `Enter`: emit plan to stdout and exit 0
- `q` / `Ctrl+C`: exit 1 (cancel, emit nothing)
- `?`: help overlay
- `x`: split (phase 2, leave as no-op initially)

## Slash command flow
1. User runs `/commit-compose` (default range = commits ahead of upstream, fallback `HEAD~10..HEAD`).
   - Variants: `/commit-compose HEAD~5`, `/commit-compose base..head`.
2. Slash command (markdown file) instructs Claude to:
   - Resolve the range
   - Refuse if working tree is dirty
   - Refuse if any commit in range is already pushed to a protected branch (config-driven list, default `main`, `master`)
   - Launch the `commit-composer` binary with the range as args
   - Read structured plan from binary stdout
   - Show plan to user, ask confirmation
   - Apply via `GIT_SEQUENCE_EDITOR=<helper> git rebase -i <base>` where the helper writes the generated todo
3. On rebase conflict: stop, surface conflict, let the user resolve and `git rebase --continue`.

## Output contract (binary stdout on Enter)
```
## commit-composer plan v1
base: <full-sha>
range: <base>..<head>
ops:
- pick   a1b2c3d
- squash d4e5f6a
- reword 789abc0 :: new commit message here
- drop   0fedcba
- pick   1234567
```
Parsing is line-based, tolerant of trailing whitespace. Reword messages use ` :: ` as the separator; multi-line messages base64-encoded after a `b64::` prefix.

Empty stdout or exit code != 0 means "cancelled, do nothing".

## Implementation phases
1. **M1 viewer**: launch TUI, parse range, render commit list + details + diff. Read-only.
2. **M2 actions**: per-commit Action state, key bindings, status line, plan serialization on Enter.
3. **M3 rebase**: implement `internal/git/rebase.go` - generate todo, run `git rebase -i` with `GIT_SEQUENCE_EDITOR`.
4. **M4 slash command**: write `commands/commit-compose.md`, end-to-end test against a throwaway repo.
5. **M5 reorder + reword UX**: J/K moves, modal input for reword.
6. **M6 polish**: terminal detection from revdiff, help overlay, README, install script.
7. **M7 optional split**: hunk-level selection inside a commit. Punt if scope creeps.

## Testing
- `internal/plan`: table-driven serialize/parse round-trip tests
- `internal/git`: integration tests that `git init` a temp repo, create fixture commits, exercise log/diff/rebase wrappers (use `t.TempDir()`)
- `internal/tui`: `github.com/charmbracelet/x/exp/teatest` snapshot tests for key flows
- E2E: a shell script in `test/` that scripts a known rebase scenario and asserts post-rebase git log

## Safety rules (must enforce in slash command)
- Abort if working tree dirty (`git diff --quiet && git diff --cached --quiet`)
- Abort if any commit in range exists in `origin/main` or `origin/master`
- Always show the proposed todo to the user and require explicit confirmation before running rebase
- Never run `git push --force` from the plugin; print the suggested push command for the user to run manually
- On rebase failure, never `git rebase --abort` automatically - surface the conflict and stop
